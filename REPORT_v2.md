# doc_extractor — Технический отчёт v2

> Сервис извлечения данных из инвойсов (document_code 04021) на базе LangExtract
> Дата: 2026-02-24
> Статус: MVP завершён — 217/217 позиций на инвойсе 206447

---

## 1. Обзор проекта

**doc_extractor** — это Django REST-сервис, реализующий ветку `document_code == 04021` (счета-фактуры / инвойсы) из воркфлоу `document_extractor_optimized`.

Сервис принимает сырой OCR-текст документа, прогоняет его через 5-шаговый пайплайн с использованием LLM через LangExtract и возвращает структурированный JSON с извлечёнными позициями инвойса, метриками качества и историей запросов.

**Текущие результаты (job_id 29):**
- 217 / 217 позиций на инвойсе 206447
- Средняя заполненность полей: ~99.8%
- Время: 190 секунд (Gemini 2.5 Pro, 1 чанк, платный ключ)
- Fallback не использовался

### 2.2 Пайплайн (5 шагов)

```
OCR Text
   │
   ▼
[Step 1] clean_text()
   Нормализация пробелов, добавление CURRENCY DATABASE в начало текста.
   Параллельно:
   ├── extract_header()        → первые 25 непустых строк как шапка
   └── parse_header_metadata() → regex-парсинг номера/даты/отправителя
   │
   ▼
[Step 2] extract_with_langextract()   ← primary model
   additional_context = header        → шапка в каждый чанк
   max_char_buffer = 32000            → весь инвойс = 1 чанк
   max_workers = 10                   → параллельная обработка чанков
   │
   ▼
[Step 3] validate_and_parse()
   Парсинг JSON, восстановление обрезанного JSON,
   фильтрация пустых позиций, нормализация hs_code.
   │
   ├── [valid] ───────────────────────────────────────┐
   │                                                   │
   └── [invalid] → [Step 4] extract_with_langextract() ← fallback model
                        └── validate_and_parse() (повторно)
                                                   │
                                                   ▼
                                          [Step 5] finalize_items()
                                          post_fill_from_header()
                                          cost = price / quantity (если пусто)
                                          unit = "pcs" (если пусто)
                                          Резолвинг валюты из БД
                                                   │
                                                   ▼
                                          JSON-ответ + RunMetrics
```

**Примечание:** функция `deduplicate_items()` существует в коде, но не вызывается. Из промпта также убрано правило DEDUPLICATION — заменено на "Extract ALL rows". Это позволило получить 217/217 на инвойсе 206447, где многие строки имеют одинаковые price/quantity (разные товары по одной цене).

## 5. Как работает LangExtract

**LangExtract** — open-source Python-библиотека от Google для структурированного извлечения данных из текстов с помощью LLM.

### 5.1 Чанкинг текста

LangExtract автоматически делит входной текст на части по параметру `max_char_buffer`:

```
OCR-текст (например, 28 000 символов)
         │
         ▼
   lx.extract(max_char_buffer=32000)
         │
         └── 1 чанк (весь текст помещается)
                  │
             LLM запрос
                  │
             JSON items
```

При большем тексте:

```
OCR-текст (>32 000 символов)
         │
         ▼
   lx.extract(max_char_buffer=32000)
         │
   ┌─────┴──────────────────┐
   Chunk 1    Chunk 2    Chunk 3
   (32000)    (32000)    (остаток)
      │          │          │
   LLM req   LLM req   LLM req   ← параллельно (max_workers=10)
      │          │          │
   items      items      items
   └──────────┴──────────┘
              │
        объединённый список
```

### 5.2 Параметр `additional_context`

Дополнительный контекст, который LangExtract добавляет **в начало промпта каждого чанка**:

```python
lx.extract(
    text_or_documents=context_text,
    additional_context=header_context,  # ← добавляется к каждому чанку
    max_char_buffer=32000,
)
```

В проекте в `additional_context` передаётся шапка документа (первые 25 строк OCR), которая содержит номер инвойса, дату, валюту, страну отправителя.

Без `additional_context`: чанки 2, 3, ... не видят шапку → поля метаданных остаются пустыми.
С `additional_context`: каждый чанк видит шапку → LLM заполняет метаданные для всех позиций.

### 5.3 Структура промпта, отправляемого в LLM

Для каждого чанка LangExtract формирует следующую структуру:

```
[EXTRACTION_PROMPT]
"You are an expert Invoice Translator and Data Extractor..."
"Extract every visible row that has a price..."
"Extract ALL rows — do NOT skip or deduplicate any lines..."

[Few-shot Examples]
ExampleData → показывает LLM точный формат JSON-ответа
(hs_code, description, quantity, unit, cost, price, currency, ...)

[Additional Context]
=== DOCUMENT HEADER (applies to ALL items in this invoice) ===
<первые 25 строк OCR — номер, дата, валюта, отправитель>

[Chunk Content]
=== CURRENCY DATABASE (REFERENCE) ===
[{"code":"EUR","name":"Euro"},...]

=== INVOICE CONTENT ===
<текущий чанк OCR-текста>
```

### 5.4 Формат возвращаемых данных

LangExtract возвращает объект `result` с полем `result.extractions`:

```python
result = lx.extract(...)
for extraction in result.extractions:
    item = {"description": extraction.extraction_text}
    if extraction.attributes:
        item.update(extraction.attributes)
    items.append(item)
```

- `extraction.extraction_class` — класс (`"invoice_item"`)
- `extraction.extraction_text` — основной текст (описание товара)
- `extraction.attributes` — dict со всеми остальными полями

### 5.5 Взаимодействие с LLM API

LangExtract использует LiteLLM для маршрутизации к разным провайдерам:

```
lx.extract(model_id="gemini-2.5-pro", api_key=GEMINI_KEY)
    → LiteLLM
        → Google AI Studio API  (для gemini-*)
        → OpenAI API            (для gpt-*)
        → Anthropic API         (для claude-*)
```

### 6.2 Как шапка инжектируется в чанки

**Проблема:** при нескольких чанках шапка (номер, дата, валюта, отправитель) есть только в первом. Остальные чанки её не видят → поля метаданных остаются пустыми.

**Решение — 2 уровня:**

```
Уровень 1: через LangExtract additional_context
  extract_header(context)
      → "=== DOCUMENT HEADER ===\nInvoice No: 206447\nDate: 15.01.2026\n..."
  lx.extract(..., additional_context=header)
      → LangExtract добавляет этот текст в каждый чанк

Уровень 2: программный fallback
  parse_header_metadata(header)
      → {"document_number": "206447", "document_date": "15.01.2026", ...}
  post_fill_from_header(items, header_meta)
      → заполняет пустые поля у всех позиций, которые LLM не заполнил
```

**Результат:**

| Поле | До оптимизации | После |
|------|---------------|-------|
| `document_number` | 0% | **100%** |
| `document_date` | ~12% | **100%** |
| `country_sender` | ~12% | **100%** |
| `currency_code` | ~53% | **100%** |

### 12.4 Анализ: почему job_id 19 дал 18 позиций вместо 17

В результирующем списке job_id 19 присутствуют **два варианта одной и той же позиции**:

| # | Поле | Позиция 9 | Позиция 18 (дубль) |
|---|------|-----------|-------------------|
| | `description` | `"Батарея «Robbe Modellsport RO-POWER ULTRA STEADY 2600МАН 7.4 VOLT receiver battery 2S MPX»"` | `"Батарея «Robbe Modellsport RO-POWER ULTRA STEADY 2600МАН"` ← **обрезана** |
| | `hs_code` | `"8507600000"` (компактный) | `"8507 60 000 0"` (с пробелами) |
| | `country_sender` | `"DE"` | `"Germany"` |
| | `cost` / `price` / `quantity` | 20.0 / 40.0 / 2 | 20.0 / 40.0 / 2 |

**Механизм появления дубля:**

Описание батареи в OCR-тексте длинное и скорее всего переносится на следующую строку:
```
Батарея «Robbe Modellsport RO-POWER ULTRA STEADY 2600МАН         ← конец страницы / чанка
7.4 VOLT receiver battery 2S MPX»
```

При разбивке OCR-текста на чанки граница чанков пришлась **посередине описания** этой позиции. В результате:
- **Чанк A** содержит начало описания + числовые данные → GLM-5 извлёк как самостоятельную позицию (описание обрезано на `"2600МАН"`, `country_sender="Germany"` из шапки, `hs_code` с пробелами)
- **Чанк B** содержит полное описание + числовые данные → GLM-5 извлёк как полноценную позицию (`country_sender="DE"`, компактный `hs_code`)

Дополнительные улики:
- Разный формат `hs_code` у двух копий → позиции пришли из разных чанков
- `country_sender`: `"Germany"` (полное название, из шапки) vs `"DE"` (ISO-код) — разный контекст при обработке
- `country_origin = "Неизвестно"` у всех позиций job_id 19 — GLM-5 в этом запуске не вывел страну из шапки, в отличие от job_id 13

## 13. Edge Cases — список и рекомендации

### 12.1 OCR-текст

| Edge Case | Текущее поведение | Рекомендация |
|-----------|-----------------|-------------|
| Полностью пустой `ocr_draft` | `clean_text()` → `""` → HTTP 422 | ✅ Обработан |
| Только пробелы/переносы строк | Аналогично пустому | ✅ Обработан |
| OCR с артефактами (символы заменены) | LLM частично справляется | Добавить предобработку: regex-замена типичных OCR-ошибок |
| Перевёрнутый/зеркальный текст | Пустой результат | Добавить детектор: если `<5` числовых токенов → ранний возврат ошибки |
| Многостраничный документ (склеенный) | LangExtract чанкует — работает | Проверить при `>200K символов` |
| Документ без строк с ценой | `validate_and_parse()` → empty → HTTP 422 | ✅ Обработан |

### 12.2 Структура инвойса

| Edge Case | Текущее поведение | Рекомендация |
|-----------|-----------------|-------------|
| Двуязычный инвойс (RU + EN, дубли) | `deduplicate_items()` отключена → дубли попадут | Включать `deduplicate_items()` только при детектировании дублей |
| Итоговая строка "TOTAL: 15000 EUR" | LLM иногда извлекает как позицию | Добавить фильтр: description содержит "total/итого/сумма" без quantity |
| Строки-заголовки секций без цены | `validate_and_parse()` фильтрует по отсутствию price | ✅ Обработан |
| Строки со скидкой (отрицательная цена) | LLM извлекает, нет валидации | Добавить флаг для отрицательных `price` |
| Нулевые строки (Frachtkosten, price=0.00) | Фильтруется как пустая позиция | ✅ Обработан |
| Несколько инвойсов в одном OCR | Всё извлечётся с смешанными метаданными | Добавить детектор смены шапки документа |
| Кредит-нота / credit note | Извлекается как обычный инвойс | Добавить детекцию типа документа |

### 12.3 Числовые данные

| Edge Case | Текущее поведение | Рекомендация |
|-----------|-----------------|-------------|
| Европейский формат: `1.200,50` | Правило в промпте → LLM нормализует | ✅ Работает |
| Числа с пробелами: `1 200 000` | LLM нормализует | ✅ Работает |
| Quantity отсутствует (подразумевается 1) | `cost = price / 1 = price` | ✅ Обработан |
| Отрицательные числа (скидки, кредиты) | LLM извлекает без валидации | Добавить флаг в метрики |
| NaN или бесконечность (повреждённый OCR) | LLM обычно пропускает строку | Добавить защиту в `finalize_items()` |

### 12.4 Поля метаданных

| Edge Case | Текущее поведение | Рекомендация |
|-----------|-----------------|-------------|
| Нет номера инвойса в OCR | LLM извлекает похожее или null | Добавить `confidence` флаг |
| Несколько дат (инвойс + поставка) | LLM берёт первую дату | Добавить `delivery_date` как отдельное поле |
| Страна отправителя только в адресе | `parse_header_metadata()` regex не найдёт | Расширить regex: детекция по городу/индексу |
| Несколько валют в одном инвойсе | Разные `currency_code` для разных строк | ✅ Поддерживается (поле на уровне позиции) |
| HS код без точек: `8471300000` | Передаётся как есть | Добавить нормализацию: `8471.30.00.00` |
| Неполный HS код (4 знака): `8471` | Передаётся как есть | Добавить валидацию минимальной длины |
| Страна происхождения числовым кодом: `752` | LLM может не распознать | Добавить словарь ISO numeric → ISO alpha |

### 12.5 LLM-поведение

| Edge Case | Текущее поведение | Рекомендация |
|-----------|-----------------|-------------|
| Markdown-обёртка ````json...```` | `validate_and_parse()` стрипает | ✅ Обработан |
| Обрезанный JSON (токен-лимит) | `rfind("}")` recovery | ✅ Обработан |
| Ответ `{"items": [...]}` вместо `[...]` | Нормализация через `.get("items")` | ✅ Обработан |
| Python-литералы `True`/`None` | `_repair_json()` исправляет | ✅ Обработан |
| Trailing comma `[{...},]` | `_repair_json()` убирает | ✅ Обработан |
| LLM галлюцинирует HS коды | Нет проверки | Добавить валидацию против справочника ТН ВЭД |
| LLM переводит описания на EN | Редко (Gemini 2.5 Pro), чаще на Flash | Добавить постпроцессинг-детектор смены языка |
| API timeout | HTTP 500 | Добавить retry с exponential backoff |
| Rate limit 429 | LangExtract падает с ошибкой | Добавить retry-обёртку в `extract_with_langextract()` |

## 14. Известные ограничения

### 13.1 Квота API
- Бесплатный ключ Gemini: 5 RPM, 20 RPD — только для разработки
- Платный ключ снимает RPM-лимит
- При `max_char_buffer=32000` инвойс 206447 = 1 чанк = 1 API-вызов

### 13.2 Fallback-модель
- Fallback (GPT-4o) требует ключ OpenAI или работу через отдельный ключ
- Текущая реализация: оба LLM используют один `LANGEXTRACT_API_KEY`

### 13.3 Синхронное выполнение
- HTTP-запрос ждёт завершения пайплайна (20–200 секунд)
- Для production рекомендуется Celery + Redis

### 13.4 SQLite
- Не подходит для concurrent-запросов в production
- Рекомендуется PostgreSQL

### 13.5 Дедупликация — требует ручного управления
- `deduplicate_items()` нужна для двуязычных инвойсов (RU + EN = одни товары дважды)
- Вредна для инвойсов с повторяющимися числовыми характеристиками у разных товаров
- Текущее решение: выключена глобально
- Требуется: умная маршрутизация на основе детектора структуры документа

### 16.1 Быстрые улучшения (🟢 30 мин — 1 час, `extraction.py` / `currency.py` / `metrics.py`)

#### Нормализация `hs_code` в `finalize_items()`

**Проблема:** GLM-5 возвращает `"8507 60 000 0"` и `"8507600000"` непредсказуемо в зависимости от чанка (наблюдалось в job_id 15 и 19).

**Решение:** постобработка в `finalize_items()` — удалить пробелы, точки и дефисы из всех `hs_code` после извлечения.

```python
def normalize_hs_code(raw: str) -> str:
    if not raw:
        return raw
    return re.sub(r"[\s.\-]", "", str(raw))
```

Вызывать для каждой позиции в `finalize_items()`. Результат: единый формат `"8507600000"` независимо от модели и размера чанка.

---

#### Программный вывод `country_origin_code` в `finalize_items()`

**Проблема:** `country_origin_code` заполнен только в job_id 13 (случайно угадал GLM-5). В трёх других запусках — 0%, несмотря на наличие `country_origin`.

**Решение:** lookup-словарь в коде, не надеяться на LLM:

```python
COUNTRY_NAME_TO_NUMERIC = {
    "germany": 276, "deutschland": 276, "германия": 276,
    "china": 156, "китай": 156,
    "sweden": 752, "швеция": 752,
    "japan": 392, "япония": 392,
    "france": 250, "франция": 250,
    "netherlands": 528, "нидерланды": 528,
    "usa": 840, "united states": 840, "сша": 840,
    # ... расширяется по мере появления новых стран в инвойсах
}

def infer_country_origin_code(country_origin: str) -> int | None:
    if not country_origin:
        return None
    return COUNTRY_NAME_TO_NUMERIC.get(country_origin.strip().lower())
```

В `finalize_items()`: если `country_origin_code` пусто → применить `infer_country_origin_code(item["country_origin"])`. Ожидаемый эффект: fill rate `country_origin_code` вырастет с 0% до ~100% на всех запусках где `country_origin` заполнен.

---

#### Фильтр значений-сентинелей (`"Неизвестно"`) как пустых

**Проблема:** `country_origin = "Неизвестно"` считается заполненным (`fill_rate=1.0`), но семантически поле пустое. job_id 19 — 100% fill rate по `country_origin`, хотя реальная ценность = 0.

**Решение:** в `compute_field_fill_rates()` (файл `metrics.py`) и в `post_fill_from_header()`:

```python
EMPTY_SENTINELS = {"неизвестно", "unknown", "n/a", "none", "не указано", ""}

def is_filled(value) -> bool:
    if value is None:
        return False
    return str(value).strip().lower() not in EMPTY_SENTINELS
```

`post_fill_from_header()` должен применять заполнение не только к `None`, но и к значениям-сентинелям — чтобы программный fallback из шапки перезаписал `"Неизвестно"`.


### 16.2 Средние улучшения (🟡 3–4 часа, новая логика в `extraction.py`)

#### Умная дедупликация частичных дублей на границах чанков

**Проблема:** job_id 19 — батарея извлечена дважды: с полным и обрезанным описанием, потому что граница чанка пришлась посередине строки. Стандартная `deduplicate_items()` не поможет (разные строки не совпадают точно).

**Решение:** функция `smart_deduplicate_cross_chunk(items)`, вызываемая в Step 5 **только если чанков > 1**:

```python
def smart_deduplicate_cross_chunk(items: list[dict]) -> list[dict]:
    """
    Удаляет частичные дубли с границ чанков.
    Признак дубля: одинаковые price + quantity + currency,
    и одно description является началом другого (prefix match).
    Из двух кандидатов оставляет тот, у которого description длиннее.
    """
    result = []
    used = set()
    for i, a in enumerate(items):
        if i in used:
            continue
        merged = False
        for j, b in enumerate(items[i + 1:], start=i + 1):
            if j in used:
                continue
            same_price = a.get("price") == b.get("price")
            same_qty   = a.get("quantity") == b.get("quantity")
            same_curr  = a.get("currency_code") == b.get("currency_code")
            if same_price and same_qty and same_curr:
                da = a.get("description", "") or ""
                db = b.get("description", "") or ""
                if da and db and (da.startswith(db) or db.startswith(da)):
                    kept = a if len(da) >= len(db) else b
                    result.append(kept)
                    used.add(i)
                    used.add(j)
                    merged = True
                    break
        if not merged:
            result.append(a)
    return result
```

**Важно:** активировать только при `chunks_count > 1` — иначе можно удалить легитимные позиции с одинаковой ценой (как было на инвойсе 206447).

---

#### Перекрытие чанков (chunk overlap)

**Проблема:** корень всех дублей на границах — LangExtract делит текст «вслепую» по символьному лимиту, не учитывая границы строк таблицы.

**Решение:** при `max_char_buffer < len(text)` применять ручной сплиттинг с перекрытием перед передачей текста в LangExtract:

```python
def split_with_overlap(text: str, chunk_size: int, overlap: int = 500) -> list[str]:
    """Разбивает текст на чанки с N символами перекрытия на границах."""
    chunks = []
    start = 0
    while start < len(text):
        end = min(start + chunk_size, len(text))
        chunks.append(text[start:end])
        if end == len(text):
            break
        start += chunk_size - overlap
    return chunks
```

При `overlap=500` каждый чанк видит последние 500 символов предыдущего → описание товара на границе гарантированно попадает в один из чанков целиком. Компромисс: небольшой рост вероятности дублей (устраняется п. выше — `smart_deduplicate_cross_chunk`).
