# doc_extractor — Технический отчёт v2

> Сервис извлечения данных из инвойсов (document_code 04021) на базе LangExtract
> Дата: 2026-02-24
> Статус: MVP завершён — 217/217 позиций на инвойсе 206447

---

## 1. Обзор проекта

**doc_extractor** — это Django REST-сервис, реализующий ветку `document_code == 04021` (счета-фактуры / инвойсы) из воркфлоу `document_extractor_optimized`.

Сервис принимает сырой OCR-текст документа, прогоняет его через 5-шаговый пайплайн с использованием LLM через LangExtract и возвращает структурированный JSON с извлечёнными позициями инвойса, метриками качества и историей запросов.

**Текущие результаты (job_id 29):**
- 217 / 217 позиций на инвойсе 206447
- Средняя заполненность полей: ~99.8%
- Время: 190 секунд (Gemini 2.5 Pro, 1 чанк, платный ключ)
- Fallback не использовался

### 2.2 Пайплайн (5 шагов)

```
OCR Text
   │
   ▼
[Step 1] clean_text()
   Нормализация пробелов, добавление CURRENCY DATABASE в начало текста.
   Параллельно:
   ├── extract_header()        → первые 25 непустых строк как шапка
   └── parse_header_metadata() → regex-парсинг номера/даты/отправителя
   │
   ▼
[Step 2] extract_with_langextract()   ← primary model
   additional_context = header        → шапка в каждый чанк
   max_char_buffer = 32000            → весь инвойс = 1 чанк
   max_workers = 10                   → параллельная обработка чанков
   │
   ▼
[Step 3] validate_and_parse()
   Парсинг JSON, восстановление обрезанного JSON,
   фильтрация пустых позиций, нормализация hs_code.
   │
   ├── [valid] ───────────────────────────────────────┐
   │                                                   │
   └── [invalid] → [Step 4] extract_with_langextract() ← fallback model
                        └── validate_and_parse() (повторно)
                                                   │
                                                   ▼
                                          [Step 5] finalize_items()
                                          post_fill_from_header()
                                          cost = price / quantity (если пусто)
                                          unit = "pcs" (если пусто)
                                          Резолвинг валюты из БД
                                                   │
                                                   ▼
                                          JSON-ответ + RunMetrics
```

**Примечание:** функция `deduplicate_items()` существует в коде, но не вызывается. Из промпта также убрано правило DEDUPLICATION — заменено на "Extract ALL rows". Это позволило получить 217/217 на инвойсе 206447, где многие строки имеют одинаковые price/quantity (разные товары по одной цене).

## 5. Как работает LangExtract

**LangExtract** — open-source Python-библиотека от Google для структурированного извлечения данных из текстов с помощью LLM.

### 5.1 Чанкинг текста

LangExtract автоматически делит входной текст на части по параметру `max_char_buffer`:

```
OCR-текст (например, 28 000 символов)
         │
         ▼
   lx.extract(max_char_buffer=32000)
         │
         └── 1 чанк (весь текст помещается)
                  │
             LLM запрос
                  │
             JSON items
```

При большем тексте:

```
OCR-текст (>32 000 символов)
         │
         ▼
   lx.extract(max_char_buffer=32000)
         │
   ┌─────┴──────────────────┐
   Chunk 1    Chunk 2    Chunk 3
   (32000)    (32000)    (остаток)
      │          │          │
   LLM req   LLM req   LLM req   ← параллельно (max_workers=10)
      │          │          │
   items      items      items
   └──────────┴──────────┘
              │
        объединённый список
```

### 5.2 Параметр `additional_context`

Дополнительный контекст, который LangExtract добавляет **в начало промпта каждого чанка**:

```python
lx.extract(
    text_or_documents=context_text,
    additional_context=header_context,  # ← добавляется к каждому чанку
    max_char_buffer=32000,
)
```

В проекте в `additional_context` передаётся шапка документа (первые 25 строк OCR), которая содержит номер инвойса, дату, валюту, страну отправителя.

Без `additional_context`: чанки 2, 3, ... не видят шапку → поля метаданных остаются пустыми.
С `additional_context`: каждый чанк видит шапку → LLM заполняет метаданные для всех позиций.

### 5.3 Структура промпта, отправляемого в LLM

Для каждого чанка LangExtract формирует следующую структуру:

```
[EXTRACTION_PROMPT]
"You are an expert Invoice Translator and Data Extractor..."
"Extract every visible row that has a price..."
"Extract ALL rows — do NOT skip or deduplicate any lines..."

[Few-shot Examples]
ExampleData → показывает LLM точный формат JSON-ответа
(hs_code, description, quantity, unit, cost, price, currency, ...)

[Additional Context]
=== DOCUMENT HEADER (applies to ALL items in this invoice) ===
<первые 25 строк OCR — номер, дата, валюта, отправитель>

[Chunk Content]
=== CURRENCY DATABASE (REFERENCE) ===
[{"code":"EUR","name":"Euro"},...]

=== INVOICE CONTENT ===
<текущий чанк OCR-текста>
```

### 5.4 Формат возвращаемых данных

LangExtract возвращает объект `result` с полем `result.extractions`:

```python
result = lx.extract(...)
for extraction in result.extractions:
    item = {"description": extraction.extraction_text}
    if extraction.attributes:
        item.update(extraction.attributes)
    items.append(item)
```

- `extraction.extraction_class` — класс (`"invoice_item"`)
- `extraction.extraction_text` — основной текст (описание товара)
- `extraction.attributes` — dict со всеми остальными полями

### 5.5 Взаимодействие с LLM API

LangExtract использует LiteLLM для маршрутизации к разным провайдерам:

```
lx.extract(model_id="gemini-2.5-pro", api_key=GEMINI_KEY)
    → LiteLLM
        → Google AI Studio API  (для gemini-*)
        → OpenAI API            (для gpt-*)
        → Anthropic API         (для claude-*)
```

### 6.2 Как шапка инжектируется в чанки

**Проблема:** при нескольких чанках шапка (номер, дата, валюта, отправитель) есть только в первом. Остальные чанки её не видят → поля метаданных остаются пустыми.

**Решение — 2 уровня:**

```
Уровень 1: через LangExtract additional_context
  extract_header(context)
      → "=== DOCUMENT HEADER ===\nInvoice No: 206447\nDate: 15.01.2026\n..."
  lx.extract(..., additional_context=header)
      → LangExtract добавляет этот текст в каждый чанк

Уровень 2: программный fallback
  parse_header_metadata(header)
      → {"document_number": "206447", "document_date": "15.01.2026", ...}
  post_fill_from_header(items, header_meta)
      → заполняет пустые поля у всех позиций, которые LLM не заполнил
```

**Результат:**

| Поле | До оптимизации | После |
|------|---------------|-------|
| `document_number` | 0% | **100%** |
| `document_date` | ~12% | **100%** |
| `country_sender` | ~12% | **100%** |
| `currency_code` | ~53% | **100%** |

## 13. Edge Cases — список и рекомендации

### 12.1 OCR-текст

| Edge Case | Текущее поведение | Рекомендация |
|-----------|-----------------|-------------|
| Полностью пустой `ocr_draft` | `clean_text()` → `""` → HTTP 422 | ✅ Обработан |
| Только пробелы/переносы строк | Аналогично пустому | ✅ Обработан |
| OCR с артефактами (символы заменены) | LLM частично справляется | Добавить предобработку: regex-замена типичных OCR-ошибок |
| Перевёрнутый/зеркальный текст | Пустой результат | Добавить детектор: если `<5` числовых токенов → ранний возврат ошибки |
| Многостраничный документ (склеенный) | LangExtract чанкует — работает | Проверить при `>200K символов` |
| Документ без строк с ценой | `validate_and_parse()` → empty → HTTP 422 | ✅ Обработан |

### 12.2 Структура инвойса

| Edge Case | Текущее поведение | Рекомендация |
|-----------|-----------------|-------------|
| Двуязычный инвойс (RU + EN, дубли) | `deduplicate_items()` отключена → дубли попадут | Включать `deduplicate_items()` только при детектировании дублей |
| Итоговая строка "TOTAL: 15000 EUR" | LLM иногда извлекает как позицию | Добавить фильтр: description содержит "total/итого/сумма" без quantity |
| Строки-заголовки секций без цены | `validate_and_parse()` фильтрует по отсутствию price | ✅ Обработан |
| Строки со скидкой (отрицательная цена) | LLM извлекает, нет валидации | Добавить флаг для отрицательных `price` |
| Нулевые строки (Frachtkosten, price=0.00) | Фильтруется как пустая позиция | ✅ Обработан |
| Несколько инвойсов в одном OCR | Всё извлечётся с смешанными метаданными | Добавить детектор смены шапки документа |
| Кредит-нота / credit note | Извлекается как обычный инвойс | Добавить детекцию типа документа |

### 12.3 Числовые данные

| Edge Case | Текущее поведение | Рекомендация |
|-----------|-----------------|-------------|
| Европейский формат: `1.200,50` | Правило в промпте → LLM нормализует | ✅ Работает |
| Числа с пробелами: `1 200 000` | LLM нормализует | ✅ Работает |
| Quantity отсутствует (подразумевается 1) | `cost = price / 1 = price` | ✅ Обработан |
| Отрицательные числа (скидки, кредиты) | LLM извлекает без валидации | Добавить флаг в метрики |
| NaN или бесконечность (повреждённый OCR) | LLM обычно пропускает строку | Добавить защиту в `finalize_items()` |

### 12.4 Поля метаданных

| Edge Case | Текущее поведение | Рекомендация |
|-----------|-----------------|-------------|

## 14. Известные ограничения

### 13.1 Квота API
- Бесплатный ключ Gemini: 5 RPM, 20 RPD — только для разработки
- Платный ключ снимает RPM-лимит
- При `max_char_buffer=32000` инвойс 206447 = 1 чанк = 1 API-вызов

### 13.2 Fallback-модель
- Fallback (GPT-4o) требует ключ OpenAI или работу через отдельный ключ
- Текущая реализация: оба LLM используют один `LANGEXTRACT_API_KEY`

### 13.3 Синхронное выполнение
- HTTP-запрос ждёт завершения пайплайна (20–200 секунд)
- Для production рекомендуется Celery + Redis

### 13.4 SQLite
- Не подходит для concurrent-запросов в production
- Рекомендуется PostgreSQL

### 13.5 Дедупликация — требует ручного управления
- `deduplicate_items()` нужна для двуязычных инвойсов (RU + EN = одни товары дважды)
- Вредна для инвойсов с повторяющимися числовыми характеристиками у разных товаров
- Текущее решение: выключена глобально
- Требуется: умная маршрутизация на основе детектора структуры документа

## 15. Дальнейшие рекомендации

### 14.1 Краткосрочные (высокий приоритет)

| Задача | Обоснование |

| Celery + Redis | Асинхронная обработка, клиент получает `job_id` мгновенно |

| Другие ветки `document_code` | Сейчас реализована только ветка `04021` |
| Кэширование результатов | Одинаковый OCR → один результат, повторный LLM-вызов не нужен |
| Webhook / callback | Уведомление DocAi о завершении вместо поллинга |
| Health-check эндпоинт | `GET /api/health/` — проверка ключа, БД, доступности LLM |
